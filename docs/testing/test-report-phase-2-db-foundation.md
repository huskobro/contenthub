## Issues Encountered

| Issue | Fix |
|-------|-----|
| `setuptools.backends.legacy` not found on Python 3.9 | Changed `build-backend` to `setuptools.build_meta` |
| `Optional[str]` required — Python 3.9 does not support `str \| None` syntax at runtime | Used `from typing import Optional` and `Mapped[Optional[str]]` |
| `sqlite3.OperationalError: unable to open database file` | Created `backend/data/` directory (alembic runs from `backend/`, relative path resolves there) |
| `ModuleNotFoundError: No module named 'greenlet'` | Added `greenlet>=3.0.0` to pyproject.toml dependencies and installed |
| WAL mode not shown by plain `sqlite3` connection | Confirmed via async SQLAlchemy engine — pragma event listener fires correctly per connection |

## Known Limitations / Deferred

- `backend/data/` must exist before running alembic from the `backend/` directory. A `backend/data/.gitkeep` should be committed to handle fresh checkouts.
- `greenlet` is now a runtime dependency (not dev-only) because SQLAlchemy async requires it at import time.
- No model-level unit tests for `AuditLog` and `User` — covered implicitly by table existence check. Full CRUD tests for these will be added when their service layers are built.

---

## Decision Trail

- WAL mode chosen for SQLite: better concurrent read performance, standard for local-first apps.
- Foreign key enforcement: disabled by default in SQLite — must be explicitly enabled per connection.
- `aiosqlite` driver: only async SQLite driver compatible with SQLAlchemy 2.0 async API.
- Three bootstrap tables only: domain models (settings, jobs, templates, etc.) deferred to their respective phases per CLAUDE.md phased delivery order.
