# ContentHub — Runtime & Storage Policy

**Version:** 1.0  
**Date:** 2026-04-20  
**Status:** FINAL

This document defines the authoritative boundary between source code,
runtime state, and artifact storage. Every file in the repository must
be classifiable under one of the four categories below.

---

## Four Categories — Hard Rules

| Category | Lives in git? | Example |
|---|---|---|
| **Source code** | YES | `backend/app/**`, `frontend/src/**`, `alembic/versions/**` |
| **Runtime state** | NO — local only | `backend/data/contenthub.db`, WAL/SHM files |
| **Job artifacts** | NO — local only | `backend/workspace/**` |
| **Local-only tooling** | NO — ignored | `.claude/worktrees/**`, `.venv/`, `node_modules/` |

---

## backend/data/ — Runtime Database

### Final Decision: RUNTIME-ONLY. Not tracked in git.

**Rationale:**  
The SQLite database is operational state, not source code. Committing
it would mean git diffs contain binary DB changes on every app run,
and clones would receive stale pre-seeded data that conflicts with
fresh Alembic migrations.

**Rules:**
- `backend/data/contenthub.db` — runtime, git-ignored
- `backend/data/contenthub.db-shm` — SQLite WAL shared memory, git-ignored
- `backend/data/contenthub.db-wal` — SQLite WAL journal, git-ignored
- `backend/data/*.bak`, `*.broken-*`, `*.pre-*` — manual snapshots, git-ignored
- `backend/data/.gitkeep` — the only tracked file; ensures the directory
  exists after `git clone` so the app can write its DB immediately

**Fresh setup flow:**
```bash
git clone <repo>
cd ContentHub/backend
python -m alembic upgrade head   # creates contenthub.db from schema
python -m app.db.seed            # seeds initial data (admin user, known settings)
```

**Do NOT:**
- Commit `contenthub.db` or any `.db-shm`/`.db-wal` files
- Commit backup snapshots (`.bak`, `.broken-*`, `.pre-*`)
- Restore a stash or old commit to "recover" DB state — use Alembic

## backend/workspace/ — Job Artifacts

### Final Decision: RUNTIME-ONLY. Not tracked in git.

**Rationale:**  
Job workspace contains audio files, rendered video, subtitle files, JSON
manifests, and intermediate outputs — all generated by the job engine at
runtime. They are large, binary, and fully regenerable by rerunning jobs.
Committing them bloats the repo and makes diffs meaningless.

**Rules:**
- `backend/workspace/**` — all content git-ignored
- `backend/workspace/.gitkeep` — the only tracked file; ensures the
  directory exists after clone

**What lives inside (runtime-generated, ignored):**
```
backend/workspace/{job-uuid}/artifacts/
    audio/scene_N.mp3          # TTS-generated audio
    audio_manifest.json        # audio timing manifest
    script.json                # narration script
    metadata.json              # video metadata
    composition_props.json     # Remotion render props
    render_props.json          # render config snapshot
    subtitle_metadata.json     # subtitle positions
    subtitles.srt              # subtitle file
    word_timing.json           # word-level timing
    output.mp4                 # final rendered video
    visuals/item_N_img_N.jpg   # downloaded/generated visuals

backend/workspace/users/{user-slug}/jobs/{job-uuid}/    # user-scoped jobs
backend/workspace/users/{user-slug}/exports/{uuid}/     # published exports
```

**Do NOT:**
- Add any workspace path to `.gitignore` negation rules to force-track
- Commit binary media files under any workspace path
- Treat workspace content as source of truth for job state (DB is SoT)

## data/ (root-level) — Legacy Stub

### Final Decision: STUB — only .gitkeep tracked.

`data/.gitkeep` is tracked. Everything else under root `data/` is
git-ignored by `data/*.db` rules. This directory was a legacy
placeholder; active runtime DB is at `backend/data/`.

---

## renderer/test-renders/ — Render Test Outputs

### Final Decision: RUNTIME-ONLY. Not tracked in git.

Test render outputs (PNG frames, JSON scene descriptors) are
artifact outputs from renderer development runs. They are
git-ignored via `.gitignore → renderer/test-renders/`.

---

## .claude/ — Claude Code Internal Tooling

### Final Decision: LOCAL-ONLY. Partially ignored.

`.claude/worktrees/` contains Claude Code linked worktrees — separate
git working directories managed by Claude Code's internal tooling.
These are local state that should not be committed.

`.claude/projects/` contains session memory and conversation files —
also local-only.

**Ignored paths:**
```
.claude/worktrees/
.claude/projects/
```

**Note on `?` in `git status`:** Git-registered linked worktrees
(shown with `+` in `git branch -vv`) appear as `?` in `git status`
rather than `??` because git itself manages them as internal working
tree entries. This is normal and cannot be suppressed further without
removing the worktree registration. These are not untracked files.

## .gitignore Policy Summary

Root `.gitignore` covers:
- macOS artifacts (`.DS_Store`)
- Python cache (`__pycache__/`, `.venv/`, `.ruff_cache/`)
- Node artifacts (`node_modules/`, `dist/`, `coverage/`)
- Secrets (`.env`, `*.pem`, `credentials.json`)
- Root-level data stubs (`data/*.db`)
- **Backend runtime DB** (`backend/data/*.db`, `*.db-shm`, `*.db-wal`)
- **Backend workspace artifacts** (`backend/workspace/**`)
- Renderer test outputs (`renderer/test-renders/`)
- Claude Code tooling (`.claude/worktrees/`, `.claude/projects/`)

`backend/.gitignore` mirrors the backend-specific rules locally for
clarity when working inside the `backend/` directory.

## Source Code Boundaries — What IS Tracked

| Area | Tracked | Not Tracked |
|---|---|---|
| `backend/app/**` | ALL source files | — |
| `backend/alembic/**` | ALL migration files | — |
| `backend/tests/**` | ALL test files | — |
| `backend/scripts/**` | ALL utility scripts | — |
| `backend/data/` | `.gitkeep` only | `*.db`, `*.db-shm`, `*.db-wal`, `*.bak`, `*.pre-*` |

| `frontend/public/**` | ALL static assets | — |
| `docs/**` | ALL documentation | — |
| `renderer/**` | ALL renderer source | `test-renders/` |
| `.gitignore`, `backend/.gitignore` | YES | — |
| `CLAUDE.md`, `README.md` | YES | — |

## Clone & First-Run Checklist

After `git clone`:

```bash
# Backend setup
cd backend
python -m venv .venv
source .venv/bin/activate
pip install -e ".[dev]"
alembic upgrade head          # ← initializes DB from schema
python -m app.db.seed         # ← seeds admin user + known settings

# Frontend setup
cd ../frontend
npm install
npm run dev

# Backend dev server
cd ../backend
uvicorn app.main:app --reload
```

The following directories will be created at runtime (not from git):
- `backend/data/contenthub.db` — created by Alembic
- `backend/workspace/` — created by job engine on first job run
